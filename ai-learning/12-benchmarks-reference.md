# LLM / Agent 评估基准：数据集、指标与范式

> 秋招面试高频考点。面试官会问：
> - "你怎么评估你的模型/Agent效果？"
> - "你知道哪些评测数据集？分别测什么？"
>
> **最后更新：2025 年初知识**

---

## 〇、2024-2025 年关键变化（面试加分项）

```
2023 年看 MMLU、GSM8K、HumanEval           ← 简单但已饱和
2024 年看 MMLU-Pro、GPQA、SWE-bench         ← 更难 + Agent 基准兴起
2025 年看 SWE-bench Verified、τ-bench、A2A  ← 更可靠 + 多 Agent 协作
```

**三大趋势**：

1. **基准全面饱和 → 换更难的**
   - GSM8K 人人 95%+，不再有区分度 → 换 MATH / AIME / GPQA / MMLU-Pro
   - MMLU 也已接近天花板（闭源 90%+），MMLU-Pro 和 GPQA Diamond 成为新标尺
   - HumanEval 基本满分 → 换 BigCodeBench / SWE-bench

2. **Agent 评估崛起**
   - 从「模型单次回答」转向「多步推理 + 工具调用 + 长时间运行」的 Agent 评测
   - 真实环境评测（WebArena、OSWorld、Computer Use）成为 Agent 面试高发题
   - 新增评估维度：工具调用准确率、步骤效率、错误恢复率、端到端任务完成率

3. **评测可靠性受重视**
   - 旧基准数据污染严重 → SWE-bench Verified、LiveBench 等防污染方案出现
   - LLM-as-Judge 被广泛接受但需注意风格偏好和长度偏差
   - Chatbot Arena（众包 ELO）成为公认最权威的真人偏好排行

## 一、业界做评估的四种主流范式

### 范式1：学术基准评测（标准化排行榜）

这是"考大学"式的评测——所有模型在同样的题目上比分数。

```
                    排行榜
                 ┌────┴────┐
             闭源模型      开源模型
           GPT-4o: 88.7%   Llama-3: 86.1%
           Claude: 88.0%   Qwen-2: 85.5%
```

**特点**：
- 标准化、可复现
- 测的是模型本身的能力，不是你的业务效果
- 主流榜单：Open LLM Leaderboard、LMSYS Chatbot Arena、SuperGLUE

**适用场景**：选基座模型时参考，面试时展示行业知识

### 范式2：业务定制评测

这是"招聘"式的评测——你的Agent能不能干好你的活。

```
你有一个客服Agent：
        ↓
收集1000条真实用户对话
        ↓
人工标注：标准答案 + 正确的工具调用序列
        ↓
跑你的Agent，看它能做对多少
        ↓
输出：完成率 87%，平均2.3轮解决问题
```

**特点**：
- 与业务强相关
- 需要人工标注，成本高
- 最真实反映Agent在业务上的表现

**适用场景**：你的项目实战，简历中的核心论据

### 范式3：LLM-as-Judge 自动评测

用一个更强的LLM来给你的Agent打分。

```
用户问题 ──→ 你的Agent ──→ 回答
                              │
                              ↓
                     GPT-4/Claude 评分
                              │
                              ↓
                         分数 + 原因
```

**特点**：
- 自动化、低成本、可大规模跑
- Chatbot Arena（众包 + ELO 评分）已成为公认最权威的真人偏好排行榜
- AlpacaEval 2、Arena-Hard、MT-Bench 是常用自动评测替代方案
- 核心风险：Judge 有风格偏好和长度偏好，高分≠真的好——需要长度控制或多 Judge 交叉验证

**适用场景**：快速迭代时的对比测试，特别是 A/B 测试哪个回复更好

### 范式4：真实环境评测（2024-2025 新增）

在真实或仿真环境中执行任务，而不是只输出一段文字。

```
WebArena:     在真实网站上完成购物/订票任务
OSWorld:      Ubuntu/Windows 桌面操作
SWE-bench:    修 GitHub Issue → 跑 CI → 看测试通过率
Computer Use: 点击真实 GUI（按钮、输入框、菜单）
τ-bench:      多轮对话 + 工具调用的长期任务
```

**特点**：
- 最真实、最难、最贵
- 评估的不只是"会不会说"，更是"会不会做"
- 2025 年 Agent 面试的高频考点

**适用场景**：Agent 产品上线前的最终验证

## 二、评估指标：三层体系（面试重点）

### 第1层：模型能力指标（测LLM本身）

| 指标 | 定义 | 典型数据集 | 2025 年 SOTA 参考 |
|------|------|-----------|-----------------|
| **准确率** | 选择题正确率 | MMLU, MMLU-Pro | o1 ~92% (MMLU)，~75% (MMLU-Pro) |
| **Pass@k** | 生成 k 次，至少一次通过测试 | HumanEval, BigCodeBench | HumanEval 已饱和（人人>90%），BigCodeBench 取代之 |
| **数学推理** | 最终答案正确率 | GSM8K, MATH, AIME | GSM8K 饱和(~95%)；MATH o1 ~94%, DeepSeek-R1 ~91%；AIME o1 ~74% |
| **研究生级推理** | 更难的科学推理 | GPQA Diamond | 人类专家 ~65%, o1-preview ~73%（首次超人类） |
| **指令遵循** | 是否严格遵守格式/约束 | IFEval, FollowBench | ~85%+ |
| **幻觉率** | 编造事实的比例 | HaluEval, SelfCheckGPT | 越低越好，仍在持续改进 |
| **安全性** | 是否拒绝有害请求 | SafetyBench, WildJailbreak | >90% 基础线，但越狱仍是难题 |

**关键变化**：
- GSM8K、HumanEval 已饱和，不再有区分度 → 面试已不再问这两个
- 现在问的是 **MATH、GPQA Diamond、AIME、MMLU-Pro**（更难、拉得开差距）
- GPQA Diamond 是 2024 年爆火的新基准：o1-preview 是首个稳定超过人类专家的模型

**面试话术示例**：
> "选基座模型时，我会分层看：知识广度看 MMLU 和 MMLU-Pro，推理深度看 MATH 或 GPQA Diamond（研究生级别），代码能力看 BigCodeBench。比如 DeepSeek-V3 在 MATH 上做到 ~90%，GPQA Diamond ~60%，这说明推理已经接近一线水平。"

### 第2层：系统能力指标（测RAG/Agent整体）

| 指标 | 定义 | 怎么算 |
|------|------|--------|
| **任务完成率** | Agent是否达成用户目标 | 人工/LLM判断 |
| **端到端成功率** | 工具调用的序列最终返回预期结果（SWE-bench 这类） | 测试是否通过 |
| **工具调用准确率** | 工具名称+参数是否正确 | 精确匹配 / LLM-as-Judge |
| **检索精准率** | Top-K结果中相关文档比例 | `相关数 / K` |
| **检索召回率** | 召回的相关文档占应有总数 | `召回数 / 总相关数` |
| **忠实度** | 回答是否忠于检索内容不编造 | RAGAS faithfulness |
| **答案相关性** | 回答是否对用户有用 | RAGAS answer_relevancy |
| **步骤效率** | 完成任务需要多少轮 | 越少越好 |
| **错误恢复率** | 出错后能否自行纠正 | 恢复次数 / 总出错次数 |
| **上下文利用率** | 长上下文中是否准确用到了关键信息 | Needle-in-a-Haystack |
| **拒绝率** | 对不该回答的请求说"不" | 越高越好 |

**2025 年面试话术**：
> "评价 Agent 我最关注三个数：**任务完成率**（能做事）、**工具调用准确率**（会做事）、**平均回合数**（做得快）。三者缺一不可。"

### 第2.5层：Agent 专项评估基准（2025 年新增）

这些不再是"考大学"而是"考实操"——面试官一定会问到。

| 基准 | 测什么 | 怎么测 | 为什么重要 |
|------|--------|--------|-----------|
| **SWE-bench Verified** | 修真实 GitHub Issue | 跑 CI 看测试过没过 | 2025 年最火，Agent 写代码能力的金标准 |
| **GAIA** | 多步推理 + 搜索 + 工具 | 需要上网搜、用工具、多步推理才能答对 | 专测"像人一样解决问题" |
| **WebArena** | 网页上的自动化操作 | 在 Amazon/GitLab/Reddit 等真实网站完成任务 | Agent 操作网页能力 |
| **OSWorld** | 桌面操作系统操作 | 在 Ubuntu/Windows 完成文件管理等任务 | Computer Use 能力 |
| **τ-bench** | 多轮对话+工具长期任务 | 长时间多轮交互完成复杂任务 | 真实 Agent 场景，高区分度 |
| **BFCL v2** | 函数调用 / Function Calling | 函数选择+参数填写的正确率 | 工具调用的标准化评测 |

**关键变化**：
- SWE-bench 有数据污染问题 → SWE-bench **Verified** 做了净化，成为新标准
- Replit、Devin、Claude Code 等编程 Agent 都以 SWE-bench Verified 作为核心参考
- GAIA 的区分度极高：人类 > 90%，最强 AI Agent ~50-60%——说明 Agent 离人类还有巨大差距

### 第3层：工程指标（测生产可用性）

| 指标 | 说明 | 典型目标 |
|------|------|---------|
| **P50延迟** | 一半请求在多少时间内完成 | < 2s |
| **P95延迟** | 95%的请求在多少时间内完成 | < 5s |
| **吞吐量(QPS)** | 每秒能处理多少请求 | 取决于业务 |
| **Token消耗** | 每次请求平均花多少Token | 越低越好 |
| **成本** | 每万次请求的费用 | 按预算 |
| **异常率** | 报错/超时比例 | < 1% |
| **稳定性** | 连续运行是否有性能衰减 | 7×24不降 |

**面试话术示例**：
> "模型效果好但延迟高到用户无法接受也不行。我的评估会加上 P50/P95 延迟和 Token 成本，综合看准确率/成本比。比如我们用 DeepSeek 后成本降了 60%，准确率只降了 3%，这是划算的。"

## 三、公认数据集大全（面试必知）

### 3.1 通用知识与推理

| 数据集 | 测什么 | 形式 | 规模 | 面试要点 |
|--------|--------|------|------|---------|
| **MMLU** | 57学科知识广度 | 选择题 | ~14K题 | 最权威的通用知识基准 |
| **C-Eval** | 中文知识(对标MMLU) | 选择题 | ~14K题 | 中文模型必测 |
| **AGI-Eval** | 更难的多任务 | 选择题+简答 | ~8K题 | 清华出品，测推理上限 |
| **BBH** | 23个复杂推理任务 | 多种形式 | 23 tasks | BIG-Bench中的难题子集 |
| **ARC** | 科学推理 | 选择题 | ~7.7K题 | 测逻辑推理 |

**面试常问**：
> "MMLU 是目前最广泛使用的通用知识评测集，覆盖 57 个学科。中文领域看 C-Eval。如果模型在这两个上分数都高，说明知识储备扎实。"

### 3.2 数学推理

| 数据集 | 测什么 | 形式 | 规模 | 面试要点 |
|--------|--------|------|------|---------|
| **GSM8K** | 小学数学应用题 | 文字题 | 8.5K题 | ⚠️ 已饱和（SOTA ~95%），面试区分度低 |
| **MATH** | 高中数学竞赛 | 文字题 | 5K题 | 现在的主力推理基准，o1 ~94%，DeepSeek-R1 ~91% |
| **AIME** | 美国数学邀请赛（高难） | 数学题 | 历年真题 | 2025年新热门，o1 ~74%，OpenAI o3 ~96% |
| **MGSM** | 多语言GSM8K | 文字题 | 10语言 | 测多语言推理一致性 |

**面试常问**：
> "GSM8K 已经饱和，现在真正拉差距的是 MATH 和 AIME。o1 在 MATH 上做到 ~94%，在 AIME 上 ~74%。如果面试官问模型推理能力，我会主动说 MATH 的分数而不是 GSM8K，后者已经没有区分度了。"

### 3.3 代码生成

| 数据集 | 测什么 | 形式 | 规模 | 面试要点 |
|--------|--------|------|------|---------|
| **HumanEval** | Python函数补全 | 函数题 | 164题 | ⚠️ 已饱和（人人>90%），面试区分度低 |
| **MBPP** | Python编程题 | 完整程序 | 974题 | 同样趋于饱和 |
| **BigCodeBench** | 复杂多文件Python任务 | 任务型 | 1.1K题 | 取代 HumanEval 的新标准，测真实工程能力 |
| **LiveCodeBench** | 实时更新编程题 | 动态更新 | — | 防数据污染，2024-2025年新热门 |
| **SWE-bench Verified** | GitHub Issue修复 | 完整项目+CI | 净化版 | **2025最火**：Agent写代码能力金标准 |
| **FullStackBench** | 全栈开发 | 前端+后端+数据库 | — | 端到端软件开发能力 |

**面试常问**：
> "HumanEval 已饱和，面试官现在会问 BigCodeBench 或 SWE-bench Verified。SWE-bench Verified 尤其关键——测的是 Agent 能不能像真正的程序员一样修 issue、跑 CI、过测试。2025 年 Claude Code 和 Devin 都在这个榜单上竞争，SOTA 大约 ~50%。"
>
> "代码评估我建议看三个维度：BigCodeBench（能不能写），SWE-bench Verified（能不能修），LiveCodeBench（有没有背题）——三者互补。"

### 3.4 Agent / 工具调用

| 数据集 | 测什么 | 形式 | 规模 | 面试要点 |
|--------|--------|------|------|---------|
| **SWE-bench Verified** | 修真实GitHub Issue | CI通过判定 | 净化验证集 | 2025年最火的Agent基准 |
| **GAIA** | 多步推理+搜索+工具调用 | 复杂问题 | 466题 | 区分度极高：人类>90%，AI ~50-60% |
| **AgentBench** | Agent通用能力 | 8个环境 | 多场景 | OS操作+Web+数据库等 |
| **WebArena** | Web自动化 | 真实网站 | 812任务 | 测网页操作Agent |
| **OSWorld** | 桌面操作系统 | 真实OS任务 | 369任务 | 测 Computer Use 能力 |
| **τ-bench** | 多轮对话+工具长期任务 | 长时间交互 | — | 真实Agent场景，高区分度 |
| **BFCL v2** | 函数调用能力 | API调用 | 2K+ | Function Call标准化评测 |

**面试常问**：
> "如果你想说自己做了Agent，一定要知道 GAIA 和 τ-bench。GAIA 需要多步推理+工具调用才能答对，人类 >90%，最强 AI Agent ~50-60%，说明 Agent 还有很大差距。τ-bench 则测长期多轮交互中的工具使用一致性。如果面试官问'你们Agent怎么评估的'，我会说用 GAIA 验证集测端到端成功率，用 BFCL v2 测函数调用准确率。"

### 3.5 RAG 专用

| 数据集 | 测什么 | 形式 | 特点 |
|--------|--------|------|------|
| **RGB** | 检索增强生成基准 | 问答对 | 测RAG四大能力 |

### 3.6 安全性 / 对齐

| 数据集 | 测什么 |
|--------|--------|

| **SafetyBench** | 模型是否拒绝有害请求 |

## 四、如何将这些数据集用到你的项目中

### 怎么在面试中展示

**策略：不要说你跑过全部，说你跑过相关的几个**

```python
"""
面试官：你怎么评估你做的那个RAG Agent？
"""

# ❌ 差回答
"我用MMLU测了，80分。"

# ✅ 好回答
"我的评估分三层：

第一层，模型选型时参考了学术界标准。
我对比了 Qwen-2.5 和 DeepSeek-V2，
重点看了 MATH（逻辑推理）和 MMLU-Pro（专业知识深度），
因为我们的 Agent 需要复杂推理能力。

第二层，Agent 端到端评估。
我用 GAIA 验证集测了多步推理+工具调用的端到端成功率，
同时用 BFCL v2 测了函数调用的准确率。

第三层，针对业务场景自己建了测试集。
从真实对话里抽了 200 条，
人工标了标准答案和工具调用路径。
最终我们的 Agent 任务完成率 87%，
工具调用准确率 94%，平均 2.3 轮解决一个问题。

要没有这套评估体系，我根本不知道哪个版本更好。"
```

## 五、面试高频追问及回答

### Q1: "这些评测数据集有什么局限性？"

```
回答要点：
1. 数据污染——模型可能在训练时见过测试题（GSM8K/HumanEval 已饱和）。缓解：用 LiveCodeBench（实时更新）和 SWE-bench Verified（经净化审查）
2. 选择题形式过于简化——MMLU 多选题 ≠ 真实复杂推理。升级：用 MMLU-Pro、GPQA Diamond 这类更难的替代
3. Agent 长期一致性——GAIA/τ-bench 虽好，但任务数有限（GAIA 466题），长周期 Agent 的稳定性和工具幻觉仍难以充分评估
4. 学术基准不能替代业务验证——GPQA 高分 ≠ 客服好用，必须自建业务测试集
5. 静态数据集无法覆盖长尾——真实边界情况测不到
```

### Q2: "你怎么保证你的评测是公平的？"

```
回答要点：
1. 测试集与训练集隔离，检查是否有数据泄漏
2. 多次运行取平均，因为生成有随机性
3. 用不同的 Judge 做交叉验证（比如 GPT-4 评 + Claude 验证）
4. 定期人工抽检，校准自动评分
```

### Q3: "你刚才说任务完成率87%，是怎么算的？"

```
回答要点（要能说出具体方法）：
1. 每条测试用例标了"成功标准"
2. 用 LLM-as-Judge 判断是否达到标准
3. 人工抽样 20% 验证准确率
4. 最终完成率 = 达标数 / 总条数
```

## 六、一句话总结

```
做 Agent 评测 = 
1. 知道业界有啥（数据集）+ 
2. 知道看啥指标（三层体系）+ 
3. 会自己建测试集（业务场景）

面试官要的不是你背出 MMLU 有多少题，
而是你能说出：为什么选它、它测什么、有什么用。
```
